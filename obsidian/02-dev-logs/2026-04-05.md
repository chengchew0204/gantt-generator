Translate a construction project Gantt chart (Chiller & CW305 installation) from a reference image into a GanttGen Excel template. Add click-to-fill-date feature on the Gantt chart grid.

- Added click-to-fill-date feature on the Gantt chart grid:
  - When the user clicks to edit a date cell (Start Date or End Date) in the DataTable, the Gantt chart enters date-picking mode.
  - Clicking on any date column in the timeline header or body grid fills the active date field for the selected task.
  - Visual feedback: crosshair cursor and accent-colored column highlight with floating label (e.g., "Start: 2026-04-07") only appears during active date picking, not when a task is merely selected.
  - Task bar / milestone clicks stop propagation to avoid triggering date-fill on existing bars.
- Refactored date-pick activation: introduced `datePickField` state in `App.jsx`, threaded through `DataTable` and `GanttChart` as a prop. The DataTable sets it when a date cell enters edit mode; the GanttChart reads it to know which field to fill and clears it after committing.
- Fixed click-to-fill-date not working due to two race conditions:
  1. Date input `onBlur` was clearing `datePickField` before the Gantt grid click could fire. Fixed by not clearing pick mode on blur (only on explicit Enter commit or Escape cancel).
  2. Task bars and milestones had `onClick stopPropagation` which blocked grid clicks when picking. Fixed by rendering a transparent overlay rect on top of the body SVG during pick mode that catches all clicks.
- Added drag-to-reorder for task rows in the DataTable. Users can grab the grip handle on the left gutter of any non-parent row and drag it to a new position. A 2px accent-colored drop indicator shows the target position. No new dependencies added (native mouse events only).
- Fixed deselection: clicking empty space in the Gantt grid, timeline header, DataTable body, or below task rows now deselects the selected task.
- Changed default view options: Critical Path and Slack Bars are hidden by default.

## Decisions made

- Used WBS numbering (1.0, 2.1, etc.) matching the source chart rather than sequential integers, preserving the original project structure.
- Mapped the weekly bar positions from the image to calendar dates starting on Mondays with weekends skipped.
- Click-to-fill-date reuses the existing `onUpdateTask` callback which already handles auto-calculation of duration and end-date via `DateUtils`, keeping the feature consistent with manual editing in the DataTable.
- Date-pick mode is explicitly driven by `datePickField` state rather than inferred from task selection, giving precise control over when the Gantt grid is clickable.

## Session 2 — Undo/Redo Stack and Keyboard Shortcuts

### Work done

- Implemented full undo/redo stack for all task mutations.
  - Created `src/hooks/useUndoRedo.js` custom hook with a max 50-entry history stack stored in refs (no unnecessary re-renders).
  - All task mutation paths (`handleUpdateTask`, `handleUpdateTaskFields`, `handleAddTask`, `handleDeleteTask`, `handleReorderTask`) now push a snapshot to the undo stack before applying the change.
  - `handleImport` (Excel import) intentionally does not record undo history since it replaces the entire dataset.
  - Batch support for drag operations: `beginBatch`/`endBatch` ensure that an entire Gantt bar drag (mousedown to mouseup) collapses into a single undo entry, regardless of how many intermediate mousemove updates occurred.
- Added keyboard shortcuts:
  - `Ctrl+S` / `Cmd+S` triggers Excel export (Save to Excel).
  - `Ctrl+Z` / `Cmd+Z` undoes the last task mutation.
  - `Ctrl+Shift+Z` / `Cmd+Shift+Z` (and `Ctrl+Y`) redoes the last undone mutation.
  - All shortcuts use `preventDefault` to avoid browser defaults.
- Added Undo/Redo buttons to the Dashboard toolbar (between the save group and PNG export).
  - Buttons show disabled state (30% opacity) when the respective stack is empty.
  - Tooltips show the keyboard shortcut (e.g., "Undo (Ctrl+Z)").
- Build verified: single-file `dist/index.html` (792 KB) produced with no errors.

### Decisions made

- Used ref-based history stacks rather than React state to avoid re-renders on every push/pop; UI disabled state is derived on-demand via `canUndo()`/`canRedo()` getter functions.
- Capped undo history at 50 entries to bound memory usage.
- Drag batching uses a simple boolean flag in the hook; `pushState` becomes a no-op while `batching.current` is true.
- Export ref pattern (`handleExportRef`) used for the Ctrl+S handler to avoid re-registering the keydown listener when tasks/settings change.

## Session 3 — UI Polish and Save-in-Place

### Work done

- Moved undo/redo from full `ActionButton` to minimal `IconButton` (icon-only, no text label), positioned to the right of the project name in the left section of the toolbar. Much smaller footprint.
- Removed "(Ctrl+S)" text from the Save to Excel button label. Shortcut is still in the tooltip.
- Implemented save-in-place using the **File System Access API**:
  - New `fileHandle` state in `App.jsx` tracks the active file handle.
  - On import: if the browser supports `showOpenFilePicker`, uses it to get a `FileSystemFileHandle` alongside the file data. The handle is stored so subsequent saves write to the same file.
  - On save (first time / new chart): uses `showSaveFilePicker` to let the user choose where to save. The returned handle is stored for subsequent saves.
  - On save (subsequent): uses `FileSystemFileHandle.createWritable()` to overwrite the file directly with no file picker dialog.
  - Fallback for browsers without File System Access API (Firefox, Safari): uses the existing `XLSX.writeFile` download approach.
  - New utility functions in `ExcelUtils.js`: `buildWorkbook`, `saveToHandle`, `saveWithPicker`, `importWithHandle`.
- Removed `handleFileSelect` from Dashboard; the import picker logic now lives in `App.jsx`'s `handleImport`.
- Build verified: single-file `dist/index.html` (794 KB).

### Decisions made

- File System Access API is used opportunistically -- feature-detected at runtime, graceful fallback to download if unsupported.
- The suggested filename for the save picker is derived from the project name (sanitized), defaulting to `GanttGen_Project.xlsx`.
- Import via `showOpenFilePicker` captures the handle for save-in-place; import via `<input type="file">` fallback does not capture a handle (API unavailable).

### Blockers & open issues

- File System Access API is not available in Firefox or Safari. These browsers will always get the download-a-new-file behavior.

## Session 4 — Arrow Key Cell Navigation in DataTable

### Work done

- Implemented keyboard navigation between cells while editing in the DataTable.
  - **Arrow Up/Down**: commits the current cell value and moves to the same column in the row above/below.
  - **Arrow Left/Right**: commits and moves to the adjacent column in the same row. For text fields, left/right only navigate when the cursor is at the start/end of the input (so normal text editing is unaffected).
  - **Tab / Shift+Tab**: commits and moves to the next/previous column.
  - **Enter**: commits and moves down to the next row (same column).
  - **Escape**: cancels editing without committing.
- Refactored editing state: lifted `editing` boolean from individual `EditableCell` components up to a centralized `editingCell` state (`{ rowIndex, colKey }`) at the `DataTable` level. Each cell is now a controlled component that receives `editing`, `onStartEditing`, `onStopEditing`, and `onNavigate` props.
- Navigation skips non-editable cells: read-only parent aggregate fields (start/end date, progress, duration) and status dropdowns are automatically skipped when navigating.
- Blur race condition fix: a `navigatingRef` flag prevents the outgoing cell's `onBlur` handler from clearing the editing state when focus shifts to the new cell via arrow/tab navigation.
- Build verified: single-file `dist/index.html` (796 KB).

### Decisions made

- Text fields allow normal left/right arrow cursor movement; navigation only triggers when the cursor is at position 0 (left) or at the end of the value (right). Number and date fields always navigate on left/right arrows since they have no meaningful cursor position.
- `navigateCell` uses a skip loop (up to `max(rows, cols)` iterations) to jump over non-editable cells in the navigation direction, rather than stopping at the first non-editable cell.
- Row selection (`onSelectTask`) is updated in sync with cell navigation so the Gantt chart highlights the corresponding task.

## Session 6 — Revert File System Access API; Simplify Save

### Work done

- Removed the File System Access API save-in-place approach entirely.
  - The API's permission model is fundamentally incompatible with the project's goals: `createWritable()` requires re-granting write permission on every call (via `requestPermission`), which in practice shows a browser permission prompt or silently fails depending on the Chrome version and context. On `file://` protocol (the primary deployment target) the API is unavailable entirely.
  - Removed `saveToHandle`, `saveWithPicker`, `importWithHandle` from `ExcelUtils.js`.
  - Removed `fileHandle` state, `fileHandleRef`, `setFileHandle` wrapper, and `showOpenFilePicker` import path from `App.jsx`.
  - Restored `handleImport` to accept a `File` argument (simpler, works everywhere).
  - Restored `handleFileSelect` in `Dashboard.jsx` for the import button.
- Save now always uses `XLSX.writeFile()` which triggers a browser download. The filename is consistently `GanttGen-{ProjectName}.xlsx`.
- Simplified `buildWorkbook` to return just the workbook object (no longer needs to produce a `Uint8Array`).
- Build verified: single-file `dist/index.html` (794 KB) -- slightly smaller than before.

### Decisions made

- The File System Access API (`showSaveFilePicker` / `FileSystemFileHandle`) is not viable for this project because: (a) it doesn't work on `file://`, (b) write permissions are not persistent across calls without user re-confirmation, (c) browser support is inconsistent. The simpler `XLSX.writeFile` download approach is universally compatible and aligns with the project's zero-installation, any-browser philosophy.

- None.
